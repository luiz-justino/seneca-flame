## Install

```sh
npm install @seneca/flame
```

## Quick Example

```js
const Seneca = require('seneca')

Seneca()
  .use('flame', { capture: true })
```

After booting, `seneca-flame` will start to capture and aggregate data automatically.

Useful commands:

- `sys:flame,capture:boolean` — Start/stop capturing data
- `sys:flame,cmd:get,cached:boolean` — Return current flame data. The `cached` option compares if the previous call has the same value; if true, replies with `{ data: false }`
- `sys:flame,cmd:snapshot,format:string` — Generate a file with current flame data (`format` can be `html` or `json`)

## More Examples

### Debug flow

When debugging, you can quickly add `seneca-flame` into your plugins, start capturing data with `sys:flame,capture:true`, proceed the actions you need to debug, then pause the capture with `sys:flame,capture:false`, and generate an `.html` file to visualize the flamegraph with `sys:flame,cmd:snapshot,format:string`.

The motivation behind this is to improve Seneca debug tools and have a tool where we can collect trace data in a structured way.

## API

### Introduction

`seneca-flame` is a Seneca plugin created as a debug tool. When using `seneca-flame`, the plugin captures metadata generated by Seneca and aggregates it in a [flame graph](https://www.brendangregg.com/flamegraphs.html).

`seneca-flame` has deep integration with [seneca-debug](https://github.com/senecajs/seneca-debug).

### Data structure and algorithm

The flame data structure uses a tree algorithm where:

- The first (root) layer is inserted by default
- The second layer of trees are always the name of the plugins
- The third and subsequent layers are the children actions of each plugin

Each node of the tree has these values:

- `name` — The name of the action or plugin
- `value` — The mean time an action took
- `children` — A list of children to this plugin/action
- `_inner` — A set of metadata for the node
